Administrator Manual

Every admin-configurable setting in RoomSync Pro, what it controls, who can change it, and the one-time Microsoft 365 setup walkthrough. Written for whoever holds the admin role, or IT staff provisioning the Azure side of the mail/calendar integration.

Roles & permissions

Every account has exactly one role, set from the Users tab. Role gates most admin surfaces; a few actions (booking edits) are additionally checked per-record rather than by role.

admin

Full access — every settings section, Users, Buildings, Backup, Kiosk. The only role that can manage other admins.

building_admin

Buildings, Users (read-only) and Activity tabs only, scoped to their assigned buildings. Cannot bulk-import rooms or open Settings.

receptionist

No Settings access. Can transfer bookings between users and approve/reject disputes.

user

Standard staff member — booking access only, no admin surfaces.

guest

One shared read-only account. Can browse and view, cannot create, edit, or cancel anything.

Guest access

"Continue as Guest" on the login screen signs the visitor into one shared guest@system.local account — no password, no per-visitor record. A single server-side rule blocks every write request for that account, so guests can look but never touch.

Note

There is no on/off switch for this. The Guest button is always present on the login page — if you need to hide it, that's a code change, not a setting.

Booking Hours

The building-wide window bookings are allowed in. Start Time / End Time, minimum 30-minute gap.

Careful

Tightening these hours immediately trims or cancels any existing booking that now falls outside the window — the save confirmation reports exactly how many were trimmed and how many were cancelled.

Weekend

Two toggles — Saturday and Sunday — mark a day as a weekend/red date across every calendar view. Turn a day off to treat it as a regular working day.

System

Business Timezone — an IANA zone (default Asia/Jakarta) used for every booking-time calculation, scheduled job, and export timestamp across the app. Change this only if the physical office itself moves time zones, not to work around a display bug.

Auto-releases bookings nobody actually shows up for, inside a confirmation window around the start time.

FieldSetting keyWhat it does

CheckboxProduces

Devices

Sensor (ESP32)

For rooms using motion/occupancy sensors instead of a kiosk tap to confirm presence under Anti-Ghost. This tab shows the shared API token and the exact request an ESP32 should send:

POST /api/sensor/ping
X-Sensor-Token: <token from Anti-Ghost settings>
Content-Type: application/json

{ "sensor_code": "<this room's sensor code>" }

Each room's individual sensor code is generated and regenerated from the room's own row in Buildings & Rooms, not from this tab.

Records

Buildings & Rooms building_admin scoped

Full CRUD for buildings and the rooms inside them: drag to reorder rooms, upload room photos, and regenerate a room's sensor code (only shown once Sensor mode is active). Import/Export is available here for both entities.

Scoping

A building_admin only sees and edits the buildings they've been assigned, and cannot bulk-import rooms at all — that action is admin-only.

Records

Users admin write · building_admin read-only

Create, edit, and delete accounts; assign role, default building, and (for receptionists/building admins) the set of buildings they manage. Search and Import/Export are built in. A building_admin can view this tab but every write action is disabled for them.

Records

Activity Log

Read-only audit trail of cancellations, role changes, user and settings edits, and data exports. Filter by category, search, and export the whole log as Excel, PDF, or TXT.

Danger zone

"Clear All" permanently deletes every logged event and is admin-only — it asks for confirmation first, and there's no undo.

Records

Disputes admin · receptionist · building_admin

When an auto-released (ghosted) booking is disputed by its owner, it lands here under Pending. Approve reinstates the booking as confirmed; Reject leaves the auto-release standing. Resolved disputes stay visible in their own filter tab for the record.

Records

Import / Export

Users, Buildings, and Rooms each support the same three input formats: Excel (.xlsx/.xls), CSV, and a single SQL INSERT INTO statement (multi-statement SQL files aren't supported).

Users

Requires name, email, password — duplicate emails are rejected. An invalid role silently falls back to user; guest can't be imported. department is auto-created if new. default_building / assigned_buildings must match an existing building name exactly (comma-separated for multiple) — these are not auto-created, so a typo fails the row. Passwords may be plain text or an already-hashed value from a prior export.

Buildings

Only name is required. location auto-creates if new. is_active defaults to yes unless explicitly set to no/false/0.

Rooms

name, capacity, floor are required. building must match an existing building exactly — not auto-created, and this whole import is admin-only (a building_admin gets a 403). facilities is a comma-separated list; status must be active or maintenance.

Integrations

Microsoft 365 setup

Lets the app send its emails and sync new bookings to Outlook Calendar through your own Microsoft tenant, instead of the default SMTP mailer. This is a one-time Azure-side setup, done by whoever administers your Microsoft 365 tenant.

What each toggle actually needs

FieldWhere it's used
Tenant ID / Client ID / Client SecretIdentify and authenticate the app to Microsoft Graph. Secret is encrypted at rest and never shown again once saved.
Sender MailboxThe mailbox address app emails are sent from, once "Send app emails via Microsoft 365" is on.
Send app emails via Microsoft 365Switches the app's mailer from SMTP to Graph. Off by default — SMTP keeps working untouched either way.
Sync new bookings to Outlook/Teams CalendarOn booking creation, adds an event to the booked-for user's Outlook calendar (same calendar Teams reads from — this is not a chat/DM notification).
When Microsoft 365 mail is off, send viaFallback mailer used whenever the mail toggle above is off — SMTP (Gmail), Log only (dev), or Discard (testing). Chosen entirely from this dropdown, no .env edit needed.

Setup walkthrough

  1. Register the app.

    Azure Portal → Entra ID → App registrations → New registration. Choose Single tenant. No redirect URI is needed. Copy the Application (client) ID and Directory (tenant) ID from the Overview page.

  2. Create a client secret.

    Certificates & secrets → New client secret. Copy the value immediately — it's shown once. Note its expiry date so the integration doesn't silently stop working later.

  3. Grant API permissions.

    API permissions → Add a permission → Microsoft Graph → Application permissions (not Delegated) → add Mail.Send and Calendars.ReadWrite. Then click Grant admin consent — this step must be done by a Global Admin/IT admin of that tenant; without it every Graph call returns 403.

  4. Confirm a licensed mailbox exists.

    Both permissions need at least one real Exchange Online mailbox in the tenant. An App Registration alone is free and needs no license — but sending mail or creating calendar events does.

  5. Fill in the Admin UI.

    Paste Tenant ID, Client ID, Client Secret → Test Connection (confirms the OAuth token works) → fill Sender Mailbox → Send Test Email (confirms Mail.Send actually delivers) → flip on the toggles you need.

Good to know

Test Connection only proves the credentials and OAuth flow are valid — it does not confirm mail or calendar actions will work. Always run Send Test Email before flipping the mail toggle on for everyone.

Known gap

Calendar sync only runs on booking creation. Editing or cancelling a booking does not yet update or remove the corresponding Outlook event.

Moving to a different tenant

An App Registration is tied to one tenant. Moving to your production Microsoft 365 tenant means repeating steps 1–3 there — only step 5's values need re-entering in this Admin UI.
