Panduan Administrator

Semua pengaturan yang bisa diubah admin di RoomSync Pro — apa fungsinya, siapa yang boleh mengubahnya — plus panduan setup Microsoft 365 yang cukup dilakukan sekali. Ditulis untuk pemegang role admin, atau staf IT yang mengurus sisi Azure dari integrasi mail/calendar.

Mulai di sini

Peran & hak akses

Setiap akun punya tepat satu role, diatur dari tab Users. Role menentukan sebagian besar akses ke halaman admin; beberapa aksi (edit booking) dicek tambahan per-record, bukan cuma berdasarkan role.

admin

Akses penuh — semua bagian Settings, Users, Buildings, Backup, Kiosk. Satu-satunya role yang bisa mengelola admin lain.

building_admin

Hanya tab Buildings, Users (hanya-lihat), dan Activity, dibatasi ke gedung yang ditugaskan padanya. Tidak bisa bulk-import rooms atau membuka Settings.

receptionist

Tidak ada akses Settings. Bisa memindahkan (transfer) booking antar user dan menyetujui/menolak dispute.

user

Staf biasa — hanya akses booking, tidak ada halaman admin.

guest

Satu akun bersama, hanya-lihat. Bisa menjelajah dan melihat, tidak bisa membuat, mengubah, atau membatalkan apa pun.

Mulai di sini

Akses Guest

Tombol "Continue as Guest" di halaman login memasukkan pengunjung ke satu akun bersama guest@system.local — tanpa password, tanpa catatan per-pengunjung. Satu aturan di server memblokir semua request tulis (write) untuk akun ini, jadi guest bisa lihat-lihat tapi tidak bisa mengubah apa pun.

Catatan

Tidak ada saklar on/off untuk ini. Tombol Guest selalu tampil di halaman login — kalau perlu disembunyikan, itu perubahan kode, bukan setting.

Booking Hours

Jendela waktu berlaku untuk seluruh gedung di mana booking diperbolehkan. Start Time / End Time, jarak minimum 30 menit.

Hati-hati

Mempersempit jam ini langsung memangkas atau membatalkan booking yang sudah ada dan kini jatuh di luar jendela waktu — konfirmasi save akan melaporkan persis berapa yang dipangkas dan berapa yang dibatalkan.

Anti-Ghost

Otomatis melepas (release) booking yang tidak ada orangnya datang, dalam sebuah jendela konfirmasi di sekitar jam mulai.

FieldSetting keyFungsinya
Enable Anti-Ghostanti_ghost_enabledSaklar utama. Mati secara default.

Booking yang tidak dikonfirmasi otomatis dibatalkan saat jendela waktu tutup, disertai notifikasi in-app dan email ke user yang bersangkutan.

Perangkat

Sensor (ESP32)

Untuk ruangan yang pakai sensor gerak/okupansi, bukan tap kiosk, untuk konfirmasi kehadiran di fitur Anti-Ghost. Tab ini menampilkan API token bersama dan format request persis yang harus dikirim ESP32:

POST /api/sensor/ping
X-Sensor-Token: <token dari Settings Anti-Ghost>
Content-Type: application/json

{ "sensor_code": "<kode sensor ruangan ini>" }

Kode sensor tiap ruangan dibuat dan di-regenerate dari baris ruangan itu sendiri di Buildings & Rooms, bukan dari tab ini.

Data

Buildings & Rooms building_admin dibatasi

CRUD penuh untuk buildings beserta rooms di dalamnya: drag untuk mengubah urutan room, unggah foto room, dan regenerate kode sensor sebuah room (hanya tampil kalau mode Sensor aktif). Import/Export tersedia di sini untuk kedua entitas.

Cakupan akses

building_admin hanya melihat dan mengubah gedung yang ditugaskan padanya, dan tidak bisa bulk-import rooms sama sekali — aksi itu admin-only.

Data

Users admin bisa tulis · building_admin hanya-lihat

Membuat, mengubah, dan menghapus akun; mengatur role, default building, dan (untuk receptionist/building admin) kumpulan gedung yang mereka kelola. Search dan Import/Export sudah tersedia. building_admin bisa melihat tab ini tapi semua aksi tulis dinonaktifkan untuknya.

Setiap akun selain admin juga punya field NIK opsional — hanya ditampilkan dan bisa diedit di tab Users ini, tidak muncul di halaman lain. Harus unik kalau diisi, dan otomatis dikosongkan kalau akunnya dipromosikan jadi Super Admin.

Data

Activity Log

Jejak audit hanya-lihat untuk pembatalan, perubahan role, edit user dan settings, serta data export. Filter berdasarkan kategori, cari, dan export seluruh log sebagai Excel, PDF, atau TXT.

Zona bahaya

"Clear All" menghapus permanen semua event yang tercatat dan admin-only — akan meminta konfirmasi dulu, dan tidak bisa di-undo.

Data

Disputes admin · receptionist · building_admin

Kalau booking yang sudah auto-release (ghosted) di-dispute oleh pemiliknya, akan muncul di sini di bawah Pending. Approve mengembalikan booking jadi confirmed; Reject membiarkan auto-release-nya tetap berlaku. Dispute yang sudah diselesaikan tetap terlihat di tab filter Resolved sebagai catatan.

Data

Import / Export

Users, Buildings, dan Rooms sama-sama mendukung tiga format input: Excel (.xlsx/.xls), CSV, dan satu statement SQL INSERT INTO (file SQL dengan banyak statement tidak didukung).

Users

Wajib ada name, email, password — email duplikat akan ditolak. role yang tidak valid otomatis jatuh ke user; guest tidak bisa diimpor. nik opsional tapi harus unik kalau diisi, dan otomatis diabaikan untuk baris dengan role = admin. department otomatis dibuat kalau belum ada. default_building / assigned_buildings harus cocok persis dengan nama building yang sudah ada (pisahkan koma untuk lebih dari satu) — ini tidak dibuat otomatis, jadi salah ketik akan membuat baris itu gagal. Password boleh teks biasa atau nilai hash yang sudah ada dari hasil export sebelumnya.

Buildings

Hanya name yang wajib. location dibuat otomatis kalau baru. is_active default-nya ya, kecuali diisi eksplisit no/false/0.

Rooms

name, capacity, floor wajib diisi. building harus cocok persis dengan building yang sudah ada — tidak dibuat otomatis, dan seluruh import ini admin-only (building_admin akan dapat 403). facilities berupa daftar dipisah koma; status harus active atau maintenance.

Integrasi

Setup Microsoft 365

Membuat aplikasi mengirim emailnya sendiri dan menyinkronkan booking baru ke Outlook Calendar lewat tenant Microsoft milik Anda sendiri, menggantikan mailer SMTP default. Ini setup sisi Azure yang cukup dilakukan sekali, oleh siapa pun yang mengelola tenant Microsoft 365 Anda.

Apa yang dibutuhkan tiap toggle

FieldDipakai untuk
Tenant ID / Client ID / Client SecretMengidentifikasi dan mengautentikasi aplikasi ke Microsoft Graph. Secret dienkripsi saat disimpan dan tidak pernah ditampilkan lagi setelah tersimpan.
Sender MailboxAlamat mailbox tempat email aplikasi dikirim dari sana, begitu "Send app emails via Microsoft 365" nyala.
Send app emails via Microsoft 365Mengalihkan mailer aplikasi dari SMTP ke Graph. Mati secara default — SMTP tetap jalan seperti biasa apa pun kondisinya.
Sync new bookings to Outlook/Teams CalendarSaat booking dibuat, menambahkan event ke kalender Outlook milik user yang di-booking-kan (kalender yang sama dibaca Teams — ini bukan notifikasi chat/DM).
When Microsoft 365 mail is off, send viaMailer cadangan yang dipakai setiap toggle mail di atas mati — SMTP (Gmail), Log only (dev), atau Discard (testing). Dipilih langsung dari dropdown ini, tidak perlu edit .env.

Langkah setup

  1. Daftarkan aplikasinya.

    Azure Portal → Entra ID → App registrations → New registration. Pilih Single tenant. Redirect URI tidak diperlukan. Salin Application (client) ID dan Directory (tenant) ID dari halaman Overview.

  2. Buat client secret.

    Certificates & secrets → New client secret. Salin nilainya segera — hanya ditampilkan sekali. Catat tanggal kedaluwarsanya supaya integrasi tidak berhenti mendadak nanti.

  3. Berikan API permissions.

    API permissions → Add a permission → Microsoft Graph → Application permissions (bukan Delegated) → tambahkan Mail.Send dan Calendars.ReadWrite. Lalu klik Grant admin consent — langkah ini harus dilakukan oleh Global Admin/IT admin tenant tersebut; tanpa ini, setiap panggilan Graph akan balas 403.

  4. Pastikan ada mailbox berlisensi.

    Kedua permission butuh minimal satu mailbox Exchange Online asli di tenant tersebut. App Registration sendiri gratis dan tidak butuh lisensi — tapi mengirim email atau membuat event kalender butuh.

  5. Isi di Admin UI.

    Tempel Tenant ID, Client ID, Client Secret → Test Connection (memastikan token OAuth berhasil) → isi Sender Mailbox → Send Test Email (memastikan Mail.Send benar-benar terkirim) → nyalakan toggle yang dibutuhkan.

Perlu diketahui

Test Connection hanya membuktikan kredensial dan alur OAuth-nya valid — belum memastikan aksi mail atau calendar akan berhasil. Selalu jalankan Send Test Email dulu sebelum menyalakan toggle mail untuk semua orang.

Keterbatasan saat ini

Calendar sync hanya berjalan saat booking dibuat. Mengubah atau membatalkan booking belum memperbarui atau menghapus event Outlook yang bersangkutan.

Pindah ke tenant lain

Satu App Registration terikat ke satu tenant. Pindah ke tenant Microsoft 365 produksi berarti mengulang langkah 1–3 di sana — hanya nilai di langkah 5 yang perlu dimasukkan ulang di Admin UI ini.
